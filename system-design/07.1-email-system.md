# Email System (Gmail-like) — Comprehensive NALSD Reference

*Seventh in the system-design set. SMTP ingestion, an MTA queue, Bigtable-style storage, threading, and full-text search — all genuinely different subsystems wired together, each with its own crux decision.*

*Callbacks throughout: [06-web-crawler.md](./06-web-crawler.md) (partition-by-owning-entity, content-hash dedup), [03.1-distributed-kv-store-nalsd.md](./03.1-distributed-kv-store-nalsd.md) (contrasted directly in Section 4 — this is the first doc in the series to deliberately choose range-partitioning over consistent hashing, and to explain why).*

---

## 1. Requirements & Capacity Estimation

### Functional requirements
- **Receive** mail via SMTP from any sending mail server, reliably, even under load spikes.
- **Store** messages durably, organized per-user, retrievable by folder/label and by conversation thread.
- **Thread** related messages into conversations.
- **Search** the full text of a user's mailbox with near-real-time freshness (a message should be findable within seconds of arriving).
- **Send** outbound mail via SMTP relay to the recipient's mail server.

### Non-functional requirements
- **Durability is non-negotiable for delivery** — an email accepted by the system must never be silently lost, even if a downstream processing stage crashes mid-pipeline. This is the requirement that shapes the entire ingestion architecture (Section 3).
- **Massive scale, per-user partitioned** — unlike most of this series' systems, almost every access pattern here (read inbox, search, thread view) is scoped to *one user's own mail*, never cross-user. This shapes the storage design (Section 4) more than almost anything else.
- **Near-real-time search freshness** — batch indexing overnight isn't acceptable; the product expectation is "I can find the email I just received."
- **Eventual consistency is fine for some things, never for others** — a label change or read/unread flag syncing a few seconds late across devices is invisible to the user; a lost or duplicated message delivery is not. Worth stating this distinction explicitly up front, since it determines which parts of the system can be AP-style (doc 03.1's philosophy) and which can't.

### Scale target
- **500M active users**, averaging **25 messages/user/day** (received + sent combined)
```
Total messages/day: 500,000,000 × 25 = 12,500,000,000 (12.5B)
Average QPS: 12,500,000,000 ÷ 86,400 sec ≈ 144,676 msgs/sec
Peak QPS (3x burst factor): ≈ 434,000 msgs/sec
```
- **Average message size: 75 KB** (headers + body + a size-weighted share of attachments — deliberately not "just the text," since attachments dominate the byte count even though most individual messages are small).

## 2. High-Level Architecture

```
Sending MTA ──SMTP──▶ Ingestion MTA (validate, SPF/DKIM check) ──▶ Durable queue, partitioned by recipient
                                                                            │
                                        ┌───────────────────────────────────┼───────────────────────────────────┐
                                  Spam/abuse filter                  Storage writer                       Search indexer
                                  (async consumer)                (async consumer)                     (async consumer)
                                        │                                  │                                    │
                                        └──────────────────▶  Bigtable-like store (Section 4)   ◀── indexed by  ┘
                                                                     +  separate full-text index (Section 6)
```

Every arrow past the queue is an **independent asynchronous consumer** of the same durable log — this is the load-bearing architectural idea in Section 3, worth understanding before any of the individual boxes.

## 3. SMTP Ingestion + MTA Queue — Accept Fast, Durable, Process Async

### The crux decision
An SMTP-receiving MTA has two options when a message arrives: (a) synchronously do everything — spam-check, write to the recipient's mailbox, index for search — before acknowledging the sending server, or (b) durably persist the message to a queue and acknowledge immediately, doing the heavier processing asynchronously afterward.

**Option (a) is a reliability trap.** SMTP senders retry on failure or timeout, but you don't want the sending server's connection held open through an entire multi-stage pipeline — any one stage (search indexing, say) being briefly slow or unavailable would cascade into delivery failures or retried duplicate sends for something that had, in truth, already fully arrived.

**Option (b) — the standard MTA design** — is the correct one: the ingestion MTA's *only* job is to validate the message is deliverable (recipient exists, basic SPF/DKIM checks) and **durably** write it to a partitioned, replicated queue before acknowledging the sender. This is structurally the same pattern as a write-ahead log: the queue write is the durability guarantee, and everything downstream (spam filtering, storage, indexing) is a replayable, independently-retriable consumer of that log — if the storage-writer consumer crashes mid-processing, it resumes from where it left off in the queue; the message was never at risk of being lost, because it was durably queued before the SMTP sender ever got its ACK.

### Queue partitioning
The queue is **partitioned by recipient mailbox** (hash of `user_id`), not by sender or arrival order — directly the same "partition by owning entity" principle as doc 06's partition-by-host for the crawler frontier. This guarantees all of one user's incoming messages are processed **in order** within their partition (useful for correct threading and label application), while different users' messages process fully in parallel across partitions.

## 4. Storage — Bigtable-Style, Range-Partitioned by User (Not Hashed)

### The crux decision, and a deliberate contrast with doc 03.1
Every prior storage design in this series (doc 03.1's KV store, doc 06's crawler metadata) defaulted to **consistent hashing** — spreading a single entity's load across many nodes to avoid hot spots. Email storage makes the **opposite** choice, and it's worth understanding exactly why.

**Row key design**: `user_id + reverse_timestamp` (or `user_id + thread_id + timestamp` — see Section 5's trade-off). Bigtable-style stores keep rows **sorted by key**, and sorted-by-`user_id` means **all of one user's mail lives in contiguous rows** — physically collocated on the same tablet(s), not scattered.

**Why collocation is exactly right here, unlike doc 03.1 or doc 06**: the dominant access pattern for a mailbox is "give me this user's most recent N messages" — a **range scan** over one contiguous key range. If mail were hash-partitioned by message ID the way doc 03.1's KV store partitions by key, fetching one user's inbox would require a scatter-gather read across many random shards for every single inbox load — exactly the kind of request pattern that punishes hash partitioning and rewards range partitioning by natural key. This is DDIA Chapter 6's key-range-vs-hash trade-off made concrete: hash partitioning optimizes for spreading load evenly at the cost of destroying locality for range queries; range partitioning by a natural key preserves locality for exactly the query pattern this system actually has, at the cost of reintroducing the hot-partition risk that hashing exists to solve (Section 7 works this out with real numbers).

**Column families**: separate families for headers, body, labels/flags, and an **attachment pointer** — actual attachment bytes live in blob/object storage (the same lesson doc 06 established for crawled page content: large blobs don't belong in a row-oriented store sized for small values), with the row holding only a reference.

**Attachment deduplication**: the same PDF sent to 50 recipients shouldn't be stored 50 times. Content-address attachments by hash (the same content-hash technique doc 06 used for exact content-duplicate detection) — 50 messages referencing one stored blob, not 50 copies.

## 5. Threading

- **Primary signal**: RFC 5322 headers — `Message-ID` (assigned once, uniquely, to every message), `In-Reply-To`, and `References` (a chain of prior Message-IDs a reply is responding to). A new root message gets a fresh `thread_id`; a reply carrying an `In-Reply-To` pointing at a known `Message-ID` inherits that message's `thread_id`.
- **Fallback heuristic** (headers are missing or broken often enough in practice to need one — some clients mangle them, forwarded mail frequently loses the chain): normalize the subject line (strip `Re:`/`Fwd:` prefixes), then combine with participant overlap and a time-window proximity check. A secondary, lower-confidence signal, not a replacement for the header-based approach.
- **Storage implication, tied directly back to Section 4's row-key choice**: embedding `thread_id` into the row key (`user_id + thread_id + timestamp`) makes "open this conversation" an efficient contiguous range scan too — but at a cost, since a pure `user_id + timestamp` key optimizes strict recency-ordered inbox view instead. Production systems typically maintain **both** — the primary row key optimized for one access pattern, plus a secondary index (or a second column family) serving the other — rather than picking one query pattern to always be slow.

## 6. Full-Text Search

**Why this can't be the same store as Section 4**: Bigtable-style row stores are built for row-key point/range lookups, not token-based text search — there's no efficient way to ask "which rows contain the word 'invoice'" against a store organized entirely around sorted keys. This needs a genuinely separate system: an **inverted index** (Lucene/Elasticsearch-style), where the index maps tokens to the documents containing them.

- **Indexing is one more async consumer of the Section 3 queue** — not a separate pipeline bolted on afterward. A message becomes searchable as a direct consequence of the same durable-queue-plus-independent-consumers architecture that handles storage and spam filtering; there's no special-cased "search sync" step.
- **Index sharding, same principle as everywhere else in this doc**: sharded **by user**, since search is always scoped to one person's own mailbox, never cross-user. This keeps each shard a manageable, bounded size rather than building one monolithic global index — the same "partition by owning entity" idea from Sections 3 and 4, applied a third time.
- **Freshness requirement is real, not aspirational** — batch/nightly indexing would violate the stated product expectation directly (Section 1). The indexing consumer has to keep pace with the ingestion rate in near-real-time, which is exactly the kind of requirement that determines node count in Section 7 — and, as it turns out, drives it more than any other layer.

## 7. Capacity Math and the First Real Bottleneck

### Ingestion — the naive count, and what it misses
**Naive, message-count-only estimate:**
```
434,000 msgs/sec peak ÷ 50,000 msgs/sec/node (a Kafka-like broker's typical sustained throughput)
= 8.68 → 9 nodes
```
**This ignores message size entirely.** At 75 KB average, actual network bandwidth is the real question:
```
434,000 msgs/sec × 75 KB = 32,550,000 KB/sec ≈ 31.8 GB/sec aggregate ingestion bandwidth

÷ 1.25 GB/sec/node (a 10Gbps NIC)
= 25.4 → 26 nodes
```
**26 nodes, not 9 — bandwidth is the binding constraint, not message count**, by roughly a 2.9x margin. This is the same category of error as doc 03.3's fan-out miss and doc 04's missing-headroom miss: a plausible-looking first-pass formula that silently assumed the wrong dimension mattered. Every topic in this series has now found a genuinely different resource turning out to be the actual constraint — worth having that pattern-recognition instinct front and center: always check *which* dimension (count, size, address space, policy) actually binds before trusting a first-pass node count.

### Storage
```
12,500,000,000 msgs/day × 75 KB = 937,500,000,000 KB ≈ 937.5 TB/day raw
× 3 (replication) ≈ 2.81 PB/day of new storage capacity needed, continuously
```
Same treatment as doc 06's content store: this is a real, large number, but it's a **provisioning/cost question tied to retention policy**, not a throughput bottleneck in the sense the other layers have one — the fix is compression, tiered storage (hot recent mail on fast storage, cold old mail on cheaper archival tiers), and a deliberate retention policy, not "add more nodes" in the way Sections 3 or the search layer below need.

### Search indexing — the layer that actually needs the most nodes
```
434,000 msgs/sec peak ÷ 5,000 docs/sec/node (indexing throughput, incl. tokenization)
= 86.8 → 87 nodes
```
**87 nodes for search indexing alone — more than 3x the ingestion layer's 26.** Worth stating plainly since it cuts against the intuitive assumption that the "front door" (SMTP ingestion) is the expensive part of an email system: **indexing for near-real-time search is the most node-hungry layer in this entire design**, a direct, quantified consequence of Section 6's freshness requirement not being negotiable.

### The first real bottleneck: hot mailboxes on a range-partitioned store
Section 4's range-partitioning choice, made deliberately to optimize inbox reads, reintroduces exactly the hot-partition risk hash partitioning exists to avoid. **A single high-traffic address** — a corporate support alias, a mailing-list archive, a spam target — can receive **orders of magnitude more mail than a typical user**:
```
Normal user: 25 msgs/day
Hot mailbox example: 100,000 msgs/day  →  4,000x normal load, concentrated on one user's key range
```
Because that user's rows are deliberately collocated (Section 4's whole point), this load lands on whichever tablet server currently owns that key range — a real, concrete hot spot, not a hypothetical one.

**Unlike doc 06's per-host politeness ceiling — which had no engineering fix at all — this one does.** Bigtable-style stores support **automatic tablet splitting**: when a tablet exceeds a size or load threshold, it's dynamically divided at a key boundary *within* its assigned range — including within a single hot user's own row range, split further by timestamp sub-ranges. The hot mailbox's data ends up spread across multiple tablet servers without changing the overall range-partitioning philosophy or the inbox-scan access pattern that motivated it in the first place. Worth stating this contrast explicitly: **this is a bottleneck that's fixable by the storage system's own mechanics, not a policy ceiling** — a genuinely different resolution shape than doc 06's finding, even though both are "hot key" problems on the surface.

- **Elasticsearch/Lucene documentation** on inverted indexes — **free**, **supplementary** — useful background for Section 6 if asked to go deeper on how tokenization and the index structure actually work.

## 9. Summary

| Layer | Node count / figure | Binding constraint |
|---|---|---|
| SMTP ingestion (MTA/queue) | 26 nodes | **Bandwidth (31.8 GB/sec), not message count** — a 2.9x correction over the naive estimate |
| Storage (Bigtable-style) | 2.81 PB/day | Provisioning/retention cost, not throughput |
| Search indexing | **87 nodes — the largest layer** | Near-real-time freshness requirement (Section 6), not raw message volume |
| Hot mailbox / tablet hot spot | 4,000x normal load, concentrated by design | **Fixable** — automatic tablet splitting, unlike doc 06's unfixable politeness ceiling |

**What ties this topic together**: every subsystem here made a deliberate choice that traded one property for another — durability over synchronous simplicity (Section 3), read locality over even load distribution (Section 4), freshness over indexing cost (Section 6) — and each trade-off's cost shows up precisely where you'd expect once the math is actually run, not as a surprise.

## 10. Follow-up Questions

- **"How is spam filtering integrated without slowing ingestion?"** — it's just another async consumer of the Section 3 queue, same as storage and indexing; a message can be provisionally stored and searchable while spam scoring happens in parallel, with a label/flag applied after the fact rather than gating delivery on the scan completing first.
- **"How do you handle multi-device read/unread and label sync?"** — this is exactly the eventual-consistency-is-fine case flagged in Section 1's non-functional requirements: label and flag state can use a much weaker consistency model than message delivery itself, since a few seconds of cross-device staleness on "did I already read this" has no correctness consequence the way a lost message would.
- **"What happens to search index staleness if the indexing layer falls behind during a load spike?"** — the async-consumer architecture (Section 3) means the queue itself absorbs the backlog durably; indexing catches up rather than dropping messages, at the cost of temporarily violating the near-real-time freshness target — worth naming as a deliberate, bounded degradation rather than a failure, the same "graceful degradation over hard failure" instinct as doc 03.1's AP philosophy.
