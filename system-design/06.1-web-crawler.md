## 1. Requirements & Capacity Estimation

### Functional requirements
- Starting from a set of seed URLs, fetch pages, extract outbound links, and follow them — the basic crawl loop.
- **Never fetch the same URL twice** (URL-level dedup) and **recognize when different URLs point to identical or near-identical content** (content-level dedup) — two distinct problems, addressed separately in Sections 4 and 5.
- **Respect politeness**: never overwhelm a single host with concurrent requests, honor `robots.txt` disallow rules and any stated crawl-delay.
- Support re-crawling for freshness — some pages change often and need revisiting, most don't.

### Non-functional requirements
- **Scale to billions of pages** without the frontier (the queue of URLs not yet fetched) itself becoming the bottleneck.
- **Politeness is not optional** — a crawler that DOSes sites it visits gets blocked, blacklisted, or worse; this is as much a legal/reputational constraint as a technical one, and it shapes the architecture more than raw throughput does (Section 8 is where this becomes the whole point of the design).
- **Robust to crawler traps** — infinite or near-infinite URL spaces (calendar pages, session-ID query parameters, symlink loops) that would otherwise consume unbounded crawl budget on a single host.
- **Extensible** — new content types, new dedup heuristics, new prioritization signals should plug in without a redesign.

### Scale target
- **1 billion pages/month** crawl rate.
```
Average QPS: 1,000,000,000 ÷ (30 × 86,400 sec) ≈ 386 pages/sec
Peak QPS (3x burst factor): ≈ 1,158 pages/sec
```
- **Average page size: 500 KB** (HTML plus inline metadata, not counting embedded images/video fetched separately).
- **Unique URLs to remember across the crawl's lifetime (not just this month): 10 billion** — this number matters far more than the monthly rate for Section 4's dedup design, since the dedup filter has to remember *everything ever seen*, not just this month's crawl.

## 3. The URL Frontier — Balancing Priority Against Politeness

This is the first crux decision, and it exists because the frontier has **two goals that actively conflict**: pop the most *important* URL next (prioritization), while never popping two URLs for the same host too close together (politeness). A single priority queue can satisfy one of these goals but not both — the highest-priority URL at any moment might belong to a host you just fetched from a second ago.

**The standard solution (the Mercator crawler's two-level frontier design, still the reference architecture today):**

- **Front queues (F1...Fn)** — one queue per priority tier. A URL is assigned to a tier based on estimated importance (PageRank-like signal, update-frequency history, or a simpler heuristic like domain authority). Selecting which front queue to pull from next uses weighted random selection biased toward higher tiers — important pages get crawled sooner, on average, without starving lower-priority tiers entirely.
- **Back queues (B1...Bm)** — one queue **per host**, holding URLs that are ready to fetch. A **mapping table** tracks which host owns which back queue, and a per-host **timestamp of last fetch** so the scheduler knows whether that host is currently eligible.
- **The politeness-enforcing step**: URLs move from front queues into back queues, but a back queue is only eligible to be popped by a fetcher once its host's minimum inter-request delay has elapsed since the last fetch from that host. This is where prioritization (front queues) and politeness (back queues, one per host) are cleanly separated — the front queues never need to know anything about host-level timing at all.

**Politeness, framed precisely, is a per-host token bucket** — directly the same primitive as doc 02's rate limiter, just scoped to "one host" instead of "one API client," and with the same reasoning for why token bucket (not leaky bucket) fits: a host that hasn't been crawled in a while can absorb a small legitimate burst up to its politeness budget, rather than being forced into a perfectly smoothed, unnecessarily conservative rate.

## 4. URL-Level Dedup — Why a Bloom Filter, and the Actual Math

### The problem
Before enqueueing a newly discovered URL, the crawler must check "have I already seen this?" — at 10 billion unique URLs over the crawl's lifetime (Section 1), storing the full URL string for every one of them in an exact set (a hash set, or a DB table) is expensive in both memory and lookup latency at this scale.

### Why a Bloom filter fits
A **Bloom filter** is a probabilistic set membership structure: a fixed-size bit array plus several hash functions. It can produce **false positives** (says "seen" when it wasn't — an acceptable failure mode here, it just means occasionally skipping a URL that was actually new) but **never false negatives** (never says "not seen" for something that was actually seen — which would let a URL escape dedup entirely). That one-directional error guarantee is exactly the right trade for this use case: an occasional missed page is a non-issue at this scale; a re-crawled page wastes budget but never corrupts anything.

### The actual sizing math
```
m = −(n × ln p) / (ln 2)²

where:
  n = number of items expected (10,000,000,000 unique URLs)
  p = target false-positive rate (1% = 0.01)
  m = required bit array size

m = −(10,000,000,000 × ln(0.01)) / (0.6931)²
  = −(10,000,000,000 × −4.6052) / 0.4804
  = 46,052,000,000 / 0.4804
  ≈ 9.587 × 10^10 bits
  ≈ 11.2 GB
```

**A Bloom filter covering 10 billion URLs at a 1% false-positive rate fits in ~11 GB.** This is a genuinely surprising, worth-stating-explicitly result: a naive exact-set approach (even a compact one, say 40 bytes/URL average) would need `10e9 × 40 bytes = 400 GB` — the Bloom filter is roughly **35x smaller**, and comfortably fits in a single modern machine's RAM rather than requiring its own distributed store.

**Number of hash functions:**
```
k = (m/n) × ln 2 = (9.587×10^10 / 10×10^9) × 0.6931 ≈ 9.587 × 0.6931 ≈ 6.64 → 7 hash functions
```

**Practical implication:** this is small enough to run as a **local, in-process filter on each crawler worker** rather than a shared distributed service, *provided* URLs are partitioned across workers in a way that keeps each worker's filter authoritative for the URLs it's responsible for — which is exactly what the host-based partitioning in Section 6 provides.

## 5. Content-Level Dedup — A Different Problem From URL Dedup

URL-level dedup (Section 4) catches literal repeat fetches of the same URL. It does **not** catch the very common case of **different URLs serving identical or near-identical content** — mirror sites, tracking/session-ID query parameters appended to an otherwise-identical URL, or syndicated content republished across many domains.

- **Exact duplicate detection**: hash the normalized page content (e.g., SHA-256 after stripping whitespace/comments) and check that hash against previously-seen content hashes. Catches byte-identical content regardless of URL.
- **Near-duplicate detection**: exact hashing fails for pages that are *almost* identical but not byte-for-byte (a page with a rotating ad banner, a timestamp footer, or minor templating differences). Production crawlers use **shingling + MinHash** or **SimHash** — techniques that produce similar hash values for similar content, allowing a "close enough" comparison rather than requiring exact equality. Worth knowing these exist and what problem they solve, without needing to derive the full algorithm in an interview setting — flag the technique by name and what it's for.

**Worth stating as a distinct point from Section 4's Bloom filter**: URL dedup and content dedup solve genuinely different problems and typically run as separate pipeline stages — collapsing them into "one dedup step" is a common oversimplification worth avoiding explicitly.

## 6. Partitioning — Why by Host, Not by URL Hash

This is the second, and more consequential, crux decision — more consequential because getting it wrong doesn't just cost efficiency, it breaks politeness (Section 3) outright.

**The naive approach**: partition the frontier across crawler worker nodes by `hash(URL)`, the same consistent-hashing scheme doc 03 used for its KV store. **This breaks politeness.** If URLs for the same host are scattered across many different workers by hash, no single worker has a complete view of "when did we last fetch from this host" — enforcing a per-host rate limit would require cross-worker coordination on every single fetch, reintroducing exactly the kind of shared-state bottleneck the whole architecture is trying to avoid.

**The correct approach**: partition by `hash(host)`, not `hash(URL)`. **Every URL for a given host is routed to the same worker**, which means that worker alone owns the complete politeness state (last-fetch timestamp, Section 3's back queue) for that host — enforcing the rate limit becomes a purely local decision, no coordination needed at all. This is the direct structural reason host-based partitioning is correct here, not just a stylistic preference.

**Rebalancing** uses the same consistent-hashing-with-vnodes mechanism as doc 03 (Section 3 of that doc) — adding or removing crawler worker nodes only remaps a bounded fraction of hosts, not the entire frontier.

**The hot-partition risk, framed via DDIA**: *Designing Data-Intensive Applications*, Chapter 6 ("Partitioning") covers exactly this failure mode under the name **skewed workloads / hot spots** — a partitioning scheme is only as good as its key's distribution, and a few extremely large sites (imagine partitioning by host and then discovering one host has 100 million pages) can make one partition disproportionately loaded regardless of how well the hashing itself is implemented. This is the same underlying shape as every prior doc's hot-key problem (doc 02's abusive client, doc 03's popular record, doc 04's viral link) — here it's a **popular host** instead. The standard mitigation, also from DDIA Ch. 6: split an oversized partition further, e.g., sub-partition an unusually large host's own URL space across multiple workers by URL path prefix, sacrificing some of the "one worker owns this host's politeness state" simplicity only for the specific hosts large enough to need it.

## 7. Politeness Mechanics — robots.txt and Layered Enforcement

- **`robots.txt` fetching and caching**: fetched once per host, cached with its own TTL (re-fetched periodically since sites do change their crawl rules), parsed for `Disallow` rules and an optional `Crawl-delay` directive.
- **Enforcement is layered, defense-in-depth style** — the same philosophy as doc 02's rate limiter placement (edge, gateway, per-service):
  1. **Frontier-level**: the back-queue timestamp check (Section 3) is the primary enforcement point — a URL simply isn't eligible to be popped until its host's delay has elapsed.
  2. **Fetcher-pool-level**: a hard cap on maximum concurrent connections to any single host, as a second independent safeguard in case a frontier-level bug or race ever lets two fetches to the same host through simultaneously.
- **Identify honestly**: a proper `User-Agent` string naming the crawler and providing a contact URL — not a politeness *mechanism* exactly, but a real operational practice that determines whether a site's operators respond to an accidental overload with a block or with an email.

## 8. The Real Bottleneck — Politeness Bounds Completion Time, Not Throughput

Sections 4 and 6 both turned up comfortably-sized numbers (an 11GB Bloom filter, host-partitioned workers with no coordination overhead). Worth checking directly: **what actually limits this system?**

**Worker throughput check:** assume a reference fetcher node sustains ~500 pages/sec (network + basic parse bound):
```
1,158 peak pages/sec ÷ 500 pages/sec/node = 2.32 → 3 nodes
```
Trivial — throughput was never going to be the constraint, the same finding as doc 05's ID generator (raw capacity wildly exceeds the target). **The actual bottleneck is structural, not a capacity number at all: politeness bounds how fast any single host can ever be fully crawled, independent of how much total cluster capacity exists.**

**Worked example — crawling one large site completely:**
```
A large site with 60,000,000 pages, politeness delay of 1 request/sec to that host:

Time to fully crawl this one site = 60,000,000 pages × 1 sec/page
                                   = 60,000,000 seconds
                                   ≈ 694 days ≈ 1.9 years
```
**No amount of additional crawler worker nodes changes this number**, because Section 6 established that one host's entire request budget is enforced by a single per-host rate limit, and adding more workers elsewhere doesn't add capacity *for that host*. This is a fundamentally different kind of bottleneck than anything in the prior five docs — not a resource that can be scaled by adding machines, storage, or a bigger data structure, but a **policy-level ceiling** on a single dimension (per-host request rate) that engineering effort cannot route around.

**What this means for hitting the Section 1 target (1,158 pages/sec peak, in aggregate):** that target is only achievable by crawling **many different hosts in parallel**, never by trying to push past any single host's politeness ceiling. The aggregate throughput target is a statement about *breadth* (how many distinct hosts are being crawled concurrently), not *depth* (how fast one host can be exhausted) — worth stating explicitly, since it's the direct justification for why Section 6's host-based partitioning is not just safe but necessary: as long as the frontier contains enough distinct hosts to keep all worker capacity busy, no single host's ceiling becomes the aggregate bottleneck. The failure mode to watch for operationally is a frontier that's accidentally narrow (e.g., an early crawl still concentrated on a small seed set) — that's when a handful of hosts' politeness ceilings *can* become the whole system's throughput ceiling.

**Is there a fix?** Only a non-engineering one: negotiate a higher crawl rate directly with a specific site's operators (large crawlers like Googlebot do maintain such arrangements for major sites), or rely on that site's own sitemap/API if one exists. There's no architectural trick that legitimately crawls a single host faster than its own stated politeness budget allows — worth stating as an honest limitation rather than searching for an engineering workaround that doesn't exist.

## 9. Storage

- **Content store**: 1B pages/month × 500KB avg = **250 TB/month** — this is an object-storage-class problem (S3-like blob storage with lifecycle/tiering policies), not a fit for the KV store design in doc 03, which was sized around small values (~1KB), not multi-hundred-KB documents. Worth being explicit that this is a different storage primitive from prior docs, not a reuse of the same one.
- **Frontier storage**: assuming a backlog of ~10x the monthly crawl target queued at any time (10B URLs), at ~100 bytes/entry (URL + priority + host pointer): **~1 TB** — small enough to be served by a distributed queue or the same KV-store pattern from doc 03, unlike the content store.
- **Metadata store** (crawl status, last-fetch timestamp, content hash per URL): same shape and scale class as doc 03's KV store — a direct reuse of that design, not a new one.

## 11. Summary

| Dimension | Value | Binding constraint? |
|---|---|---|
| Worker throughput | 3 nodes for 1,158 peak pages/sec | No — wildly over-provisioned relative to target |
| Bloom filter memory | ~11.2 GB for 10B URLs at 1% FP rate | No — fits on a single machine, ~35x smaller than an exact set |
| Frontier storage | ~1 TB | No |
| Content storage | 250 TB/month | A real number, but a provisioning/cost question, not a bottleneck in the throughput sense |
| **Per-host politeness ceiling** | **~1.9 years to fully crawl a single 60M-page site at 1 req/sec** | **Yes — the actual constraint, and the only one in this doc with no engineering fix** |

**What makes this topic's finding distinct from the prior five**: every earlier bottleneck was something more engineering effort could move — more shards, more nodes, a wider bit field, a better-provisioned DB. This is the first one that's genuinely **not fixable by adding resources** — it's a policy ceiling on a single dimension, and the entire architecture (host-based partitioning, breadth-over-depth throughput strategy) exists specifically to work *around* that ceiling by never depending on beating it in the first place.

## 12. Follow-up Questions

- **"How do you prioritize freshness — recrawling pages that change often?"** — track observed change frequency per URL (did the content hash change since last crawl?) and feed that back into the front-queue priority tier assignment from Section 3 — pages that change daily get a higher recrawl priority than ones that haven't changed in months.
- **"How do you avoid crawler traps (infinite URL spaces)?"** — URL normalization (strip known session-ID query parameters, canonicalize trailing slashes and query-parameter order before the Section 4 dedup check even runs), a max crawl depth per host, and a per-host page cap that triggers manual review if exceeded — a proactive circuit breaker on a single host's URL space, conceptually similar to doc 04's circuit-breaker-on-DB-calls pattern, just applied to a runaway URL space instead of a runaway query volume.
- **"What's different about crawling JavaScript-heavy sites?"** — requires a headless-browser-based fetcher tier (rendering the page before extracting links) rather than a plain HTTP GET, which is far more expensive per page — worth naming as a real architectural fork (a slower, heavier fetcher path for JS-rendered content) rather than assuming one fetcher design handles everything.
