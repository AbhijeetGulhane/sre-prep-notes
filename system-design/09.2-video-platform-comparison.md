# Video Platform — Comparison Against ByteByteGo and SDI Vol 1 Ch 14

*Companion to [09-video-platform.md](./09-video-platform.md). Two sources at once since they overlap heavily: the ByteByteGo podcast/course material (CDN strategy, transcoding pipeline focus) and Alex Xu's *System Design Interview* Vol 1, Chapter 14. Note on sourcing: I don't have direct transcript access to the podcast audio — the comparison below is grounded in ByteByteGo's consistent written treatment of the same material (their course content and blog, which the podcast draws from), not a verbatim listen-through.*

---

## 1. What both sources confirm

- Object storage (S3/GCS) for video bytes, a separate database for metadata — matches doc 09's Section 5 exactly.
- FFmpeg-based transcoding into multiple resolutions, feeding adaptive bitrate streaming via HLS/DASH — matches doc 09's Section 4.
- An asynchronous processing pipeline decoupled from the upload response — matches doc 09's Section 3 accept-fast-durable-queue-process-async pattern directly.
- Resumable/chunked uploads for large files — matches doc 09's Section 3.
- CDN for playback — matches doc 09's Section 7.

The core architecture lines up closely. The genuine value here is in a few specific refinements neither source's framing made obvious until you look at them directly.

---

## 2. Real gaps worth pulling into doc 09

### a) Video processing as an explicit DAG, not just "parallel renditions"
SDI Ch 14 frames the processing pipeline as a **directed acyclic graph**: some steps (metadata extraction, thumbnail generation) run once per video; others (audio/video encoding) run once *per rendition*; transcription runs once *per supported language*. Doc 09's Section 4 gets at the parallelism (across renditions, within a rendition) but doesn't frame it as a DAG with per-node independent retry — worth adopting that framing explicitly, since "which steps are per-video vs. per-rendition vs. per-language" is a cleaner mental model than a flat list of parallel jobs, and it makes the retry story (doc 09's own follow-up question about partial failure) more precise: a DAG lets one failed node retry without redoing unrelated completed work, which a flatter pipeline description doesn't make as clear.

### b) Presigned URLs — the upload service shouldn't touch the bytes at all
Doc 09's Section 3 describes an upload service that receives the raw upload directly. The more scalable, and more standard, pattern found across both sources: the upload service (control plane) authenticates the request and validates quota, then returns a **presigned, time-limited, write-scoped URL** — the client uploads the actual bytes **directly to GCS**, never routing the heavy transfer through an application server at all. This is a real **control-plane/data-plane separation** worth naming explicitly: application servers handle auth, metadata, and pipeline triggering (small, cheap operations); object storage handles the actual byte transfer (large, expensive). Doc 09's Section 8 bandwidth math (18.75 GB/sec, 15 ingestion nodes) implicitly assumed that bandwidth flows through app-tier nodes — under the presigned-URL pattern, that bandwidth bypasses the application tier entirely and hits GCS directly, meaning **those 15 "ingestion nodes" mostly disappear as a distinct cost**, replaced by a much cheaper control-plane request handling auth and URL generation. Worth revisiting Section 8's math with this correction — it's a meaningful architectural difference, not a cosmetic one.

### c) CDN usage should be selective, not blanket
Doc 09's Section 7 implicitly treats the CDN as sitting in front of all playback traffic. SDI's treatment is more precise: **CDN placement is conditional**, driven by signals like view-count thresholds, uploader subscriber count, or an ML popularity prediction — a client checks the CDN first and falls back to the origin streaming service if the content isn't there. This is a direct, concrete instance of the same hit-ratio economics doc 04's cache design worked out mathematically: blanket CDN coverage for the long tail of rarely-watched videos would be pure cost with little benefit, the same reasoning that justified doc 04's cache-aside pattern only caching what's actually being read. Worth adding this as an explicit refinement to Section 7 rather than leaving CDN coverage implicitly universal.

### d) Video deduplication — a genuine, complete gap in doc 09
Neither Section 5 nor any other part of doc 09 addresses this at all, and it's a real, well-studied problem: a meaningful fraction of uploaded video is duplicate or near-duplicate content (re-uploads, re-encodes, minor edits of existing videos). The technique mirrors doc 06's content-dedup section almost exactly, just applied to video instead of crawled text:
- **Exact-duplicate detection**: a cryptographic hash (SHA-1/SHA-256 style) of the raw file catches byte-identical re-uploads — cheap, fast, a good first-pass filter, but it fails the moment *anything* changes at the byte level (different encoding, resolution, minor edit), even if the content is visually identical to a human viewer.
- **Near-duplicate detection**: this is where video needs its own technique, distinct from doc 06's text-shingling/MinHash approach — **perceptual video fingerprinting**, extracting content-based features across sampled frames (not exact pixel values) into a compact fingerprint, then comparing fingerprints via Hamming distance rather than exact match. Conceptually the same *purpose* as doc 06's SimHash (similar content → similar hash, unlike a cryptographic hash where one changed bit produces a completely different output), just built from video frame features instead of text shingles.
- **Where this fits in doc 09's pipeline**: naturally as one more DAG node (per point (a) above) — run early, before the expensive transcoding step, so a detected duplicate can potentially skip re-transcoding entirely and reference the existing renditions instead, which is also a real cost-saving opportunity worth naming given Section 8 already established transcoding as the dominant compute cost in this whole system.

## 3. Numbers — a scoping difference, not a disagreement
SDI Ch 14's back-of-envelope estimate uses **10s of millions of DAUs, 20M daily watchers, 200,000 daily uploads, ~2GB/video average → ~400TB/day total storage** — a deliberately smaller scope than doc 09's 500-hours/minute, ~1.62PB/day raw figures. Worth reading this the same way as the 01.3 comparison's finding: **which number to use depends entirely on what an interviewer actually gives you**, not on which source's assumption is "more correct." SDI's chapter explicitly scopes down (excludes live-streaming, recommendations, content moderation) — a reasonable, deliberate interview-pacing choice, not a smaller design.

---

## 4. Practical takeaway
The presigned-URL and DAG-framing gaps are the two most worth internalizing before a real interview — both are the kind of refinement an interviewer would expect after the initial high-level pitch, and both change a concrete number or diagram in doc 09 rather than just adding color commentary. The deduplication gap is worth having as a prepared answer to "what about duplicate content" specifically, since it's a complete miss in the current doc rather than a refinement of something already there.
