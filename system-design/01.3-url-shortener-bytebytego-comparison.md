# URL Shortener — Comparison Against ByteByteGo's Reference Design

*Appendix to [01.1-url-shortener-nalsd.md](./01.1-url-shortener-nalsd.md) and [01.2-url-shortener-math-pass.md](./01.2-url-shortener-math-pass.md). Source: ByteByteGo's "Design A URL Shortener" chapter (bytebytego.com/courses/system-design-interview/design-a-url-shortener), same content as the ~15-min YouTube walkthrough — Alex Xu & Sahn Lam's standard reference answer for this exact question.*

**Why this comparison exists:** ByteByteGo's chapter is the closest thing to a "standard answer" for this question that exists in the interview-prep world. Where this doc's design agrees with it, that's real validation. Where it disagrees, that's worth understanding *why* rather than assuming either version is wrong — both are documented, defensible designs, and being able to argue either side is worth more in an actual interview than picking one as "correct."

## 1. Where the two designs matched almost exactly

| | ByteByteGo | 01.1 (this repo) |
|---|---|---|
| Short code length | Computed n=7 (62⁶=56.8B insufficient, 62⁷≈3.5T sufficient) against their 365B target | Same computation, same table logic, against a 100B target |
| ID generation approach | Compared hash+collision-resolution vs. Base62 conversion; picked Base62 + a unique ID generator | Same comparison, same pick, same core reasoning (collision-free by construction, not probabilistic) |
| Redirect mechanism | 301 saves server load via browser caching; 302 preserves analytics accuracy | Identical trade-off analysis |
| Read-path architecture | Cache sits between web servers and DB | Same |
| Rate limiting | Explicitly deferred to their own separate "Design A Rate Limiter" chapter | Same pattern — deferred to [02-rate-limiter-nalsd.md](./02-rate-limiter-nalsd.md) |

**The two "deferred to a separate chapter" rows are worth noting on their own** — ByteByteGo's book is structured with rate limiting and ID generation as standalone chapters that the URL shortener chapter explicitly tells the reader to go re-read rather than re-deriving. This repo's cross-referencing habit across topics 01, 02, and 05 mirrors that structure independently. That's the right instinct for an actual interview too: build on what's already established instead of re-deriving it from scratch every time it comes up.

## 2. The one real disagreement: database choice

**ByteByteGo recommends a relational database** — a plain 3-column MySQL-style table (`id`, `shortURL`, `longURL`), with "database replication and sharding are common techniques" mentioned only as a brief wrap-up scaling note, never stress-tested against the 365B-row number their own estimation section computed.

**01.1 defaults to a NoSQL/wide-column store** (Cassandra/Bigtable-style), with sharded MySQL explicitly listed as *this doc's own stated fallback* for exactly the scenario ByteByteGo picked.

This is a genuine difference in default, not an error on either side — and it's worth noticing that **each design already names the other's answer as its own legitimate alternative.** Both are real, production-proven patterns (Bitly ran on relational + Redis for a long time; other large-scale shorteners run on Cassandra). Worth having both directions ready:
- *"Why not just shard Postgres?"* — 01.1 already wrote this fallback case out.
- *"Why not relational at all?"* — 01.1's multi-region/horizontal-scaling justification is the argument ByteByteGo's chapter doesn't really make; they assert relational works at 365B rows without examining whether it actually does at that scale.

## 3. Gaps in 01.1 worth pulling in

- **Bloom filters.** ByteByteGo uses a bloom filter to avoid an expensive DB existence-check on every collision-resolution attempt in the hash-based approach. 01.1 rejected that approach entirely (Base62 counter has no collisions to check for), so the gap doesn't cost correctness — but the technique itself (a space-efficient probabilistic membership test) is a generally useful pattern worth having available for other "does this already exist" checks elsewhere in a design.
- **Predictable/sequential IDs as a named security concern.** ByteByteGo's comparison table calls this out directly as a con of Base62 conversion: sequential IDs make it easy to enumerate the next available short URL. 01.1 has the same underlying idea (the bit-shuffle mitigation), but it's a side note rather than a clearly labeled concern — worth promoting to a more explicit callout, since it's exactly the kind of thing an interviewer might probe on directly.

## 4. Where 01.1 goes further than the reference chapter

- **Hybrid 301+CDN redirect strategy** (01.1, Section 5) — ByteByteGo shows a plain 301 with no edge-caching layer at all.
- **Quantified hot-link math** — the ~56% traffic-share breaking point for a single cache node, worked out in 01.2. ByteByteGo's chapter doesn't address viral-link concentration at all.
- **Explicit consistent-hashing/vnode rationale** for the DB shard key — ByteByteGo stays at "sharding is a common technique" with no mechanism discussion.
- **Explicit node-count math** (01.2's storage-bound-vs-throughput-bound analysis) — ByteByteGo's estimation section stops at total storage (36.5TB) and QPS figures, without translating those into cluster sizing or checking which constraint actually binds.

## 5. Numbers side-by-side

| | ByteByteGo | 01.1 / 01.2 |
|---|---|---|
| Daily URL creation | 100M/day (given) | Not the starting input — 01.1 started from total corpus instead (see note below) |
| Read:write ratio | 10:1 | 100:1 |
| Total records (10yr horizon) | 365 billion | 100 billion (target, stated explicitly) |
| Peak write QPS | 1,160/sec (average, no peak factor applied) | 1,500/sec (peak, with an explicit 5x burst factor over a 317/sec average) |
| Peak read QPS | 11,600/sec (average, no peak factor) | 150,000/sec (peak, same 5x factor over a 31,700/sec average) |
| Avg record size | 100 bytes (URL only) | 500 bytes (URL + metadata) |
| Total storage | 36.5 TB (no replication factor applied) | 50 TB raw → 150 TB with an explicit 3x replication factor |

**Two things worth flagging honestly, not just glossing over as "different numbers":**
1. **ByteByteGo's own derived total (365B) is larger than 01.1's stated target (100B)**, despite both being pitched as a similar "large-scale interview" answer — worth noticing that "Google scale" isn't a fixed number even between two credible sources, and the right move in an interview is to work from whatever number the interviewer actually gives you rather than a memorized figure.
2. **ByteByteGo's QPS figures are averages with no stated peak/burst multiplier**, while 01.1 explicitly applies a 5x peak factor. This is very likely why 01.1's peak numbers look dramatically higher (150,000 vs 11,600) despite a *smaller* total corpus — it's a difference in what's being measured (average vs. provisioned peak), not a disagreement about the underlying traffic. Worth being explicit about which one you're quoting if an interviewer asks you to defend a specific number.

---

## 6. Practical takeaway: pacing, not correctness

ByteByteGo's version is close to what actually gets *said out loud* in a real 15-minute exercise — it's paced to fit the format. 01.1/01.2's depth (hot-link thresholds, node-count math, vnode mechanics) is what gets pulled out only if the interviewer specifically probes further ("how did you size that cluster," "what happens if one link goes viral"). Worth timing a cold run-through of the ByteByteGo-paced version specifically, and treating 01.1/01.2's extra material as the answer bank for follow-ups rather than the opening pitch — leading with the full math-pass depth in the first 15 minutes would blow the pacing an actual interview expects.
