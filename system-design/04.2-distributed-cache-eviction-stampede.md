# Distributed Cache — Eviction Policies & Stampede Solutions

*Companion to [04-distributed-cache-nalsd.md](./04-distributed-cache-nalsd.md) — Sections 4 and 5 of that doc introduced these at a survey level; this is the full mechanics, with the actual algorithms and the atomic Redis commands involved.*

These are two sides of one coin: **eviction decides what leaves the cache under memory pressure; stampede protection is what happens when the wrong thing leaves — or expires — at the wrong time**, and a flood of clients all try to recompute it at once.

---

## Part 1: Eviction Policies

### LRU (Least Recently Used)

**Exact LRU** — a doubly-linked list ordered by access recency plus a hash map for O(1) lookup, where every read moves its entry to the front. Correct, but **every read becomes a write** to the ordering structure — real overhead on the single hottest code path in the whole system, and a source of contention if that structure isn't sharded as finely as the data itself.

**Approximate LRU — what Redis actually implements, and why:** rather than maintain an exact global ordering, Redis samples a small number of random keys (default 5, tunable via `maxmemory-samples`) when it needs to evict, and evicts the least-recently-used **among that sample** — not the true global least-recently-used key.

**Why the approximation is good enough, quantified:** the quality of the approximation improves quickly with sample size, because you only need the sampled key to be "old enough," not "the single oldest key in the entire keyspace":
```
With sample size N, the probability the sample contains at least one key
from the oldest 10% of all keys (by access recency) is:

P(at least one) = 1 − (0.90)^N

N=5:  1 − 0.90^5  = 1 − 0.59 ≈ 41%
N=10: 1 − 0.90^10 = 1 − 0.35 ≈ 65%
N=20: 1 − 0.90^20 = 1 − 0.12 ≈ 88%
```
Small samples already have a meaningful chance of hitting a genuinely old key, and larger samples close the gap fast — but each eviction now costs proportionally more CPU to sample and compare. **The default of 5 is a deliberate point on this curve**, not an arbitrary number: it trades a real amount of eviction precision for O(1)-ish, bounded work per eviction — the same "cheap approximation beats expensive exactness on the hot path" trade the rate limiter doc's sliding-window-counter made (doc 02, Section 3).

### LFU (Least Frequently Used)

**Naive frequency counting** — increment a per-key counter on every access, evict the lowest count under pressure. **Breaks in an important way:** a key that was extremely popular last week but hasn't been touched in days still holds a huge counter value, and will *never* be evicted ahead of a genuinely new, moderately-popular key — stale popularity never fades on its own.

**The fix Redis actually uses — a probabilistic, decaying counter:**
- The counter is a small **8-bit** value (0–255), and it does **not** increment on every access once it's already large — increments become **probabilistically less likely as the counter grows** (a logarithmic-style counter). This means a key doesn't need an unbounded counter to represent "very popular," and a single burst of traffic doesn't instantly saturate it either.
- **Decay over time:** the counter is periodically reduced based on how long it's been since the key was last accessed — so popularity that isn't being reinforced by continued access actually fades, fixing the naive version's core problem.

**LRU vs LFU — when each is the right call:**

| | LRU | LFU |
|---|---|---|
| Optimizes for | Recency | Sustained frequency |
| Weak spot | A key accessed once, very recently, evicts something accessed 1000x but slightly longer ago | Needs decay logic or stale-but-formerly-popular keys never leave |
| Good fit | Access patterns where "recent" reliably predicts "will be accessed again soon" (session data, recent activity feeds) | Access patterns with a stable long-tail of "always hot" keys that shouldn't get evicted just because of a brief quiet period (popular product pages, reference data) |

### TTL — active vs lazy expiration

TTL is **orthogonal** to LRU/LFU — a key can leave the cache either because it was evicted under memory pressure (LRU/LFU's job) or because its TTL elapsed (expiration's job), whichever happens first. Two complementary mechanisms are needed, not one:

- **Lazy expiration:** when a client actually requests a key, check its TTL first — if expired, treat it as a miss (and physically remove it) at that moment. Cheap, but **does nothing for cold keys that expired and are never requested again** — that memory would sit reserved indefinitely without a second mechanism.
- **Active expiration:** a background process periodically samples a small random set of keys with TTLs (Redis: every 100ms, sample 20 keys by default), expires any that are past due, and — **adaptively** — immediately repeats the sampling pass if more than 25% of the sample was expired (a signal that there's a larger backlog of expired-but-unreclaimed memory to work through faster).

Both are needed together: lazy expiration keeps read-time correctness cheap and immediate; active expiration is what actually reclaims memory from keys nobody's asking about anymore.

## Part 2: Cache Stampede Solutions

### The problem, quantified
A single hot key expires (or is invalidated). Every one of its concurrent readers — plausibly thousands, for a genuinely hot key — misses at essentially the same instant and independently queries the backing DB to repopulate it.
```
Hot key with 10,000 concurrent readers, no stampede protection:
→ up to 10,000 simultaneous DB queries, for a single key, in a single instant
```
Recall doc 04's Section 1 capacity target: the backing DB is sized around ~50,000 ops/sec *total*, precisely because the cache is supposed to keep it from ever seeing peak traffic. **A single stampeding key can consume a fifth of the DB's entire sustainable budget by itself**, and if two or three hot keys happen to expire near-simultaneously, the DB can be pushed well past what it was ever provisioned for — from a cache-layer event, not an actual traffic spike.

### Solution 1: SETNX-based distributed lock

**Mechanics:** on a cache miss, before querying the DB, a client attempts to atomically acquire a short-lived lock for that specific key:
```
SET lock:{key} 1 NX PX 5000
```
`NX` — only set if the key doesn't already exist (this is the atomic "set if not present" semantic; the old two-step `SETNX` + separate `EXPIRE` pattern has exactly the same non-atomic-pair gap doc 02's Redis pattern flagged for `INCR`+`EXPIRE` — modern Redis's single `SET ... NX PX` command closes it the same way the Lua-wrapped rate-limit script did).
`PX 5000` — the lock auto-expires after 5 seconds even if never explicitly released, so a crashed lock-holder can't block everyone else forever.

- **If the `SET NX` succeeds**, this client won the right to repopulate the key: it queries the DB, writes the result to the cache, then deletes the lock.
- **If it fails** (another client already holds the lock), this client backs off — either polling the cache briefly until the winner's fresh value appears, or serving a slightly-stale fallback value if one's available, rather than also querying the DB.

**Result: at most one DB query for this key, no matter how many thousands of clients missed simultaneously** — the other 9,999 wait on the one in-flight fetch instead of each starting their own.

**The tuning trade-off, worth naming explicitly:** the lock TTL has to be long enough to cover the expected DB query time, or the lock expires mid-fetch and a second wave of clients starts a redundant second stampede; but too long, and if the lock-holder crashes mid-fetch, every waiting client is stuck blocking for the full remaining TTL before anyone retries. This doesn't fully disappear — it's a genuine trade-off between "risk of a second stampede" and "added worst-case latency for waiters," tuned by the TTL value, not eliminated by the pattern.

### Solution 2: Probabilistic early expiration

**The idea:** rather than every client treating a key's TTL as a hard wall that everyone hits at the same instant, give reads an increasing *probability* of proactively recomputing the value **before** it actually expires — spreading the refresh load out over time instead of concentrating it at one moment, and without needing a lock at all.

**The mechanism (the XFetch approach, used in production systems facing this at scale):**
```
Recompute early if:  now − (delta × β × ln(rand()))  ≥  expiry_time

where:
  delta = how long it took to originally compute/fetch this value
  β     = a tunable aggressiveness parameter (commonly 1)
  rand() ~ Uniform(0, 1), drawn independently by each reader
```
As `now` approaches `expiry_time`, the `ln(rand())` term makes the left side cross the threshold with increasing probability across the population of concurrent readers — but because each reader draws its own independent random number, **only a small number of the many concurrent readers "win" and trigger an early recompute in any given moment**, not all of them at once. The key that actually gets refreshed early is randomly and independently decided by each reader's own draw, which is exactly what spreads the load — there's no coordination needed, and no lock.

**A simpler variant many production systems actually reach for:** just add random jitter to the TTL itself at write time —
```
actual_ttl = base_ttl − random(0, jitter_window)
```
so that keys written around the same time don't all expire at the exact same synchronized instant even without any per-read probability calculation. Less precise than XFetch (it doesn't adapt to the actual observed compute cost `delta`), but far simpler to implement and reason about, and it solves the most common real-world cause of synchronized stampedes: a batch of keys all written (or all cache-warmed after a deploy) at roughly the same time, all sharing the same base TTL, and therefore all expiring together later.

**What this doesn't solve:** a genuinely **cold key** — brand new, first read ever, sudden massive concurrent traffic (a link goes viral the instant it's created) — has no prior `delta` or existing TTL to jitter around at all. Probabilistic early expiration only helps with *recompute* stampedes on keys that have cache history; the cold-start case still needs Solution 1's lock-based coalescing, because there's nothing to spread out probabilistically on a key's very first request.

### Solution 3: Application-layer request coalescing (single-flight)
A lighter-weight, in-process complement: within a single app server, if multiple concurrent goroutines/threads/requests ask for the same currently-uncached key, only the first one actually issues the fetch (to the cache and, on miss, to the DB); the rest wait on that one in-flight call and share its result. This doesn't replace the distributed lock (it only dedupes within one process, not across the whole fleet), but it's cheap, adds no network round trip, and catches a meaningful fraction of stampede traffic before it ever reaches the shared cache tier at all — a good first line of defense layered in front of Solution 1, not a substitute for it.

## Combined Recommendation

These aren't either/or choices — a production cache layer typically runs all of them together, each solving a different piece:
- **TTL (active + lazy) + LRU or LFU eviction** for capacity management — TTL decides *when* a key is stale, eviction decides *what* to remove *if* memory pressure forces a choice before TTL would have removed it anyway.
- **Application-layer single-flight** as the cheapest first line of defense against redundant fetches within one process.
- **SETNX-based distributed lock** as the cross-fleet backstop that guarantees at most one DB query per key-miss event, including cold-start stampedes.
- **Jittered TTLs (or full XFetch) on top** to reduce how often the lock even needs to be contended for in the first place, by spreading out *when* keys expire rather than only reacting after they already have.

## Follow-up questions

- **"What if the lock-holder is slow and many clients are waiting — do they all retry at once when it expires?"** — yes, this is a real secondary stampede risk if not handled; the standard fix is jittered backoff on the waiting clients' retry/poll interval, not a fixed retry delay, for the same reason jittered TTLs help with the primary stampede.
- **"Does approximate LRU ever perform noticeably worse than exact LRU in practice?"** — rarely in a way that matters; the sampled approach's occasional "wrong" eviction is a single suboptimal choice among millions, and the aggregate hit-rate difference versus exact LRU is small compared to the CPU cost exact LRU would add to every single read.
- **"How does this connect to the working-set-sizing math?"** — directly: eviction policy quality only matters once the cache is *close to* its memory limit; how often that happens, and what the resulting hit-rate degradation looks like, is exactly what the companion math-pass doc for this system works out numerically.
