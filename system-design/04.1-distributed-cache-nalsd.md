# Distributed Cache (Redis-like) вҖ” NALSD Design (Google Scale)

*Fourth in the system-design set. This one sits in front of a slower backing store rather than being the source of truth вҖ” [03-distributed-kv-store](./03-distributed-kv-store-nalsd.md) had to solve durability and conflict resolution because it *is* the durable store; this design gets to make a very different trade because losing cached data is recoverable (re-read from the DB), not catastrophic. That single difference is what justifies most of the decisions below.*

---

## 1. Requirements & Capacity Estimation

### Functional requirements
- `GET(key) -> value` / `SET(key, value, ttl?)` / `DELETE(key)` вҖ” the core contract.
- `INCR(key)` вҖ” atomic increment, the same primitive the rate limiter doc built its Redis pattern on; worth remembering this design *is* that Redis layer, viewed from the inside.
- TTL support on every key вҖ” expiration is a first-class feature here, not a workaround (contrast with the KV store, where TTL was one use case among several).
- Optional: pub/sub for cache invalidation broadcast across a fleet of app servers (Section 6).

### Non-functional requirements вҖ” the trade this design is allowed to make that the KV store wasn't
- **Sub-millisecond latency** is the entire point of this layer existing вҖ” if it's not dramatically faster than the backing DB, it has no reason to exist.
- **Losing cached data is acceptable.** A cache node crash loses that node's data; the system recovers by re-reading from the backing store on the next request (a cache miss, not data loss). This single fact is why this design can use a pure in-memory store with much weaker durability guarantees than doc 03's replicated KV store вҖ” **it doesn't need vector clocks, quorum writes, or hinted handoff, because there's always a durable source of truth one layer down.**
- **High throughput, far exceeding what the backing store could sustain directly** вҖ” this is the actual job: absorb enough read (and sometimes write) volume that the DB never sees peak traffic.
- **Approximate eviction policies are fine.** Perfect LRU is expensive to maintain at scale; a good approximation that's O(1) per operation beats an exact algorithm that adds latency to every read (Section 5).

### Scale target
Assume this cache sits in front of a general-purpose backing store (could be the shortener's DB, or any other service's) that can sustainably handle a much lower load than the traffic hitting the application.

- **Peak application traffic: 1,000,000 ops/sec**
- **Target cache hit ratio: 95%** вҖ” meaning only **50,000 ops/sec** should ever reach the backing DB at peak, a load it can actually sustain (contrast: doc 01's DB layer was sized around exactly this kind of cache-absorption ratio).
- **Working set (hot data that should stay resident in memory):** assume 250M keys, average value 2KB вҶ’ **500 GB** total hot working set.
- **Node reference class:** 64 GB usable RAM/node, ~100,000 ops/sec sustained (in-memory, single-threaded-per-shard command execution вҖ” same class of machine as the rate limiter's Redis nodes, because it's the same kind of system).

## 2. API Design

```
GET /cache/{key}        -> 200 { "value": ..., "ttl_remaining": N } | 404 (miss)
SET /cache/{key}        Body: { "value": ..., "ttl": N }  -> 200 OK
DELETE /cache/{key}     -> 200 OK  (also called "invalidate")
INCR /cache/{key}       -> 200 { "value": N }   -- atomic, same primitive as doc 02's rate limiter
```

This is almost always a **library/client-side contract**, not a public HTTP API вҖ” application code calls a Redis-protocol client directly. The HTTP shape above is just for specifying the contract precisely; the real wire protocol (Redis's RESP, or Memcached's text/binary protocol) is optimized for low per-call overhead in a way HTTP isn't, which matters at sub-millisecond latency targets.

## 3. The Crux Decision вҖ” Cache-Aside vs Write-Through vs Write-Back vs Write-Around

This plays the same role Base62-vs-MD5 played in doc 01 and token-bucket-vs-leaky-bucket played in doc 02: **the single decision that most shapes correctness and performance, and the one most worth being able to defend a specific choice on.**

### Cache-aside (lazy loading) вҖ” the default pattern
- **Read:** app checks cache first. On hit, return it. On miss, app reads the DB itself, then populates the cache for next time.
- **Write:** app writes directly to the DB, then **invalidates** (deletes) the corresponding cache entry вҖ” it does not update the cache with the new value directly.
- **Why invalidate rather than update on write:** updating the cache with the new value on every write only helps if that value gets read again soon; invalidating is cheaper and correctness-safer (a delete can't leave a half-applied or wrongly-ordered value behind the way a concurrent update can вҖ” see the race condition in Section 4).
- **Pros:** only actually-requested data ever occupies cache memory (no wasted space caching things nobody reads); the system degrades gracefully if the cache is entirely down (every read just falls through to the DB, slower but correct).
- **Cons:** every cache miss costs a full extra round trip (check cache, miss, then query DB) instead of going straight to the DB; if many concurrent requests miss the same key at once, they can all hit the DB simultaneously (cache stampede, Section 4).

### Write-through
- **Write:** app writes to the cache; the cache itself synchronously writes to the DB before acknowledging the write back to the app.
- **Read:** always served from cache вҖ” if it's been written at all, it's already there, no read-time population step needed.
- **Pros:** the cache is never stale for data that's gone through a write вҖ” there's no window where cache and DB disagree for written keys.
- **Cons:** every write pays the DB's write latency on the hot path (cache alone would be fast; this pattern deliberately gives that speed up for consistency). Also **wasteful for write-heavy-but-rarely-read data** вҖ” every write still round-trips to the DB even if the value is never read again before it expires, providing no caching benefit for that access pattern at all.

### Write-back (write-behind)
- **Write:** app writes to the cache, which acknowledges immediately; the cache asynchronously flushes to the DB later, often batched (many writes to related keys coalesced into fewer DB operations).
- **Pros:** the fastest possible write path (no DB round trip on the critical path at all), and batching can dramatically cut DB write load for high-frequency-write data (e.g., a counter incremented thousands of times a second only needs to actually persist its current value periodically, not on every increment).
- **Cons:** **real durability risk** вҖ” if the cache node crashes before its queued writes flush, those writes are lost, with no DB copy to fall back to. This is the one pattern that reintroduces something like doc 03's durability concerns, and it's an explicit, named trade-off, not an oversight: appropriate only for data where losing a few seconds of recent writes on a crash is genuinely acceptable (view counts, non-critical metrics), never for anything with real transactional meaning.

### Write-around
- **Write:** goes directly to the DB, **bypassing the cache entirely** вҖ” no cache population, no invalidation needed because the cache was never touched.
- **Read:** same lazy-population miss path as cache-aside.
- **Pros:** avoids polluting the cache with data that's written once and rarely re-read soon after (e.g., a bulk data import, or write-heavy logging data) вҖ” cache-aside or write-through would both fill precious memory with entries that provide little future hit-rate benefit.
- **Cons:** the **first** read after any write is guaranteed to be a cache miss (since the write never touched the cache), so this pattern trades a permanently-higher miss rate on just-written data for avoiding cache pollution вҖ” the right trade specifically when that data usually isn't read again immediately anyway.

### Decision matrix

| | Write latency | Read latency (after write) | Staleness risk | DB write load | Durability | Best fit |
|---|---|---|---|---|---|---|
| **Cache-aside** | Fast (DB only) | Miss once per invalidation, then fast | Brief window during the invalidation race (Section 4) | Normal | DB is source of truth, no extra risk | **General-purpose default** вҖ” most reads/writes, unpredictable access patterns |
| **Write-through** | Slow (cache + DB, synchronous) | Always fast, never a miss for written data | None for written data | Every write hits DB, even rarely-read ones | DB is source of truth | Data that's written once and read often soon after (config, session start) |
| **Write-back** | Fastest (cache only, async flush) | Fast | Cache and DB can diverge until flush | Greatly reduced via batching | **Real data-loss risk on crash** | High-frequency writes where losing the last few seconds is acceptable (counters, metrics) |
| **Write-around** | Fast (DB only) | Guaranteed miss on first post-write read | None (DB is always current, cache just lags on population) | Normal | DB is source of truth | Write-once/rarely-reread data (bulk imports, logs) |

**Recommendation: cache-aside is the correct general-purpose default** for the same reason token bucket was the general-purpose default rate-limiting algorithm вҖ” it matches the widest range of real access patterns without requiring the application to know its own read/write ratio ahead of time, and it fails safely (cache outage degrades to "slower," never to "wrong" or "lost data"). **Layer in write-through for specific hot, frequently-re-read keys** where the extra write latency is worth paying to guarantee no miss, and **write-back only for specific, explicitly-accepted-risk counters/metrics** where write volume is extreme and losing a few seconds of data on a crash is a named, accepted trade-off вҖ” not a default.

## 4. Correctness Hazards Specific to This Design

### The cache-aside invalidation race
This is a real, well-known bug class, worth walking through explicitly:
```
1. Thread A reads the DB (gets old value "v1" вҖ” a write is about to overwrite it)
2. Thread B writes "v2" to the DB, then invalidates (deletes) the cache entry
3. Thread A, still holding "v1" from step 1, now populates the cache with "v1"
```
**Result: the cache is now stuck holding the stale value "v1" until it naturally expires via TTL** вҖ” the invalidation in step 2 happened *before* the stale write in step 3, so it did nothing to prevent this. This is a genuine race, not a hypothetical: it requires no malicious timing, just two ordinary concurrent requests. Mitigations, in order of how commonly they're used:
- **Short TTLs as a safety net** вҖ” bounds how long any stale-write race can persist, even if it can't prevent it (this is the same principle as doc 01's CDN TTL choice вҖ” a bounded blast radius rather than a perfect guarantee).
- **Delete-then-write with a delay** ("double delete") вҖ” invalidate again a few seconds after the DB write, catching races that populated the cache with a stale value in that window.
- **Versioned writes** вҖ” the DB row carries a version, and the cache population step only writes to the cache if the version it read matches the current version at write time, otherwise skips populating (accepts the miss rather than risking staleness).

### Cache stampede (thundering herd)
When a hot key expires or is invalidated, many concurrent requests can all miss simultaneously and all hit the DB at once вҖ” potentially far more DB load than the DB was ever sized for, precisely because the whole point of the cache was to prevent the DB from seeing that traffic. Mitigations:
- **Request coalescing**: the first miss for a given key triggers the actual DB fetch; concurrent misses for the same key wait on that one in-flight fetch rather than each issuing their own.
- **Probabilistic early refresh**: recompute a value slightly before its TTL actually expires (with some jitter/probability), so refreshes are spread out rather than synchronized to the exact expiration moment across every client that cached it at the same time.

## 5. Eviction Policy

The cache holds far less data than the full backing store (Section 1: 500 GB working set vs. a backing store that could be far larger), so something has to decide what gets evicted under memory pressure.

- **LRU (Least Recently Used)**: evict the item that hasn't been accessed in the longest time вҖ” the standard default, matching the intuition that recently-accessed data is likely to be accessed again soon (temporal locality).
- **True LRU is expensive at scale** вҖ” it requires an exact recency ordering updated on every access, which is real overhead on the hottest possible code path. **Production systems (including real Redis) use approximate LRU**: sample a small random subset of keys, evict the least-recently-used *among the sample*, not the true global least-recently-used. This trades a small amount of eviction precision for O(1), sample-size-bounded work per eviction instead of O(log n) or worse for an exact structure вҖ” the same kind of "cheap approximation beats expensive exactness on the hot path" trade-off as the rate limiter's sliding-window-counter approximation in doc 02.
- **LFU (Least Frequently Used)**: better than LRU for access patterns with a stable "always popular" subset that shouldn't get evicted just because of a temporary quiet period вҖ” but more bookkeeping (a frequency counter, not just a recency pointer) per key.
- **TTL-based expiration** runs independently of and alongside whichever eviction policy is chosen вҖ” a key can be evicted early for memory pressure *or* expire naturally via TTL, whichever comes first.

- **Sharding via consistent hashing**, identical rationale to doc 03 вҖ” adding/removing cache nodes should only remap a small fraction of keys, not force a near-total cache-cold restart across the fleet.
- **No cross-shard coordination needed for independent keys** вҖ” unlike the KV store, this system generally doesn't need quorum writes or replication for correctness, since a lost cache node just means more cache misses temporarily, not lost data. Some deployments do add a single replica per shard purely to avoid a full cold-cache event on node loss, but it's an availability/warmth optimization, not a correctness requirement the way doc 03's N replicas were.
- **Cache is typically per-region, not globally replicated.** Unlike the KV store (where cross-region replication was an explicit availability trade-off), a cache's entire value proposition is low latency вҖ” replicating cache contents across regions would add cross-region latency to exactly the operation that's supposed to be fast, defeating the purpose. Standard practice: each region runs its own independent cache in front of its own (possibly globally-replicated) backing store, and accepts that each region's cache warms up independently.
- **Hot-key handling** вҖ” the same recurring structural issue as every prior doc in this series: one key lives on one shard regardless of cluster size. Mitigations here specifically: a small, short-TTL **local in-process cache on each app server** in front of the shared distributed cache (an L1 cache in front of an L2 cache, functionally), which absorbs the very hottest keys before they even reach the shared cache tier at all вҖ” the same principle as doc 01's CDN edge layer, just one hop earlier in the stack.

## 7. Bottlenecks & Likely Follow-Up Questions

- **"What if the cache and DB disagree and nobody notices?"** вҖ” cache-aside's failure mode is always "briefly stale, self-healing via TTL," never "silently and permanently wrong" вҖ” worth contrasting explicitly with write-back's real durability risk when asked to compare the two.
- **"How do you size the cluster if the working set doesn't fit?"** вҖ” this is exactly the numeric question the companion math-pass doc works out: what actually happens to hit ratio (and therefore backing-DB load) once the cache is undersized relative to the working set, and where that failure becomes non-linear.
- **"Persistence вҖ” should this cache survive a restart?"** вҖ” optional snapshotting (Redis's RDB) or an append-only write log (AOF) can be layered on for faster warm-up after a planned restart, but this is explicitly an optimization, not a correctness requirement, given Section 1's premise that losing cache data is always recoverable from the backing store.
- **"Why not just use the KV store from doc 03 as the cache too?"** вҖ” because doc 03 was deliberately built for durability and correctness under partition (quorum writes, vector clocks, hinted handoff) вҖ” real, valuable properties that are *pure overhead* for a workload that doesn't need them. This design's entire value is being simpler and faster by explicitly not solving problems it doesn't have вҖ” a good example of matching the tool to the actual requirement rather than reusing the most sophisticated thing already built.
