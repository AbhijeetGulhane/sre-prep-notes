# Distributed Cache — NALSD Math Pass

*Applies the numbers to [04-distributed-cache-nalsd.md](./04-distributed-cache-nalsd.md), same procedure as the prior three math passes. This one's honest finding is different in shape from the others: the cache layer itself degrades gracefully under pressure — the real fragility turns out to be one number in doc 04 that was never given any margin at all.*

---

## 0. Fixed inputs (carried from doc 04)

- Peak application traffic: **1,000,000 ops/sec**
- Target cache hit ratio: **95%** → 50,000 ops/sec designed to reach the backing DB
- Working set: 250M keys × 2KB avg value = **500 GB**
- Cache node reference class: 64 GB usable RAM/node, ~100,000 ops/sec/node
- DB node reference class: reusing **15,000 ops/sec/node** from doc 01, for consistency across this notes set

---

## 1. First-pass cache node count

**Throughput-driven:**
```
1,000,000 ops/sec ÷ 100,000 ops/sec/node = 10 nodes
```
**Memory-driven (naive, value bytes only):**
```
500 GB ÷ 64 GB/node = 7.8 → 8 nodes
```
Take the max of the two constraints: **10 nodes, throughput-bound** — same shape as doc 02's rate limiter (throughput-bound cluster), not doc 01's shortener (storage-bound).

---

## 2. Per-key overhead correction — checked, and it turns out to be small

The Section 1 memory calculation used raw value bytes only (2 KB/key) and ignored per-key bookkeeping overhead — every real in-memory store pays some fixed cost per key for the hash table entry, object wrapper, and TTL tracking, independent of the value's own size. A realistic estimate for this overhead class: **~70 bytes/key**.

```
Corrected per-key footprint: 2048 + 70 = 2118 bytes
250,000,000 × 2118 bytes = 529.5 GB  (was 500 GB — a 5.9% increase)

529.5 GB ÷ 64 GB/node = 8.27 → 9 nodes  (was 8)
```
**Worth checking, and worth reporting honestly that it barely matters here**: 9 nodes on the memory dimension is still well under the 10 nodes the throughput dimension already required, so this correction changes nothing about the binding constraint or the final node count. Contrast this explicitly with the two much larger corrections found in prior passes — the KV store's 2.2x replication fan-out (doc 03, Section 2) and the shortener's 100x memory-estimate error (doc 01, Section 2). Not every check turns up a dramatic result, and reporting a small, non-binding correction accurately is itself the right instinct — a math pass that only ever finds dramatic problems would be suspicious.

---

## 3. The number that got provisioned with no margin at all: DB capacity

Doc 04, Section 1 sized the backing DB at **exactly 50,000 ops/sec** — the precise result of `1,000,000 × (1 − 0.95)`. Stop and notice what that means before testing anything: **this is a point estimate treated as a hard capacity plan**, with zero headroom for the hit ratio ever being anything other than exactly 95%. Every prior doc in this series deliberately built in a margin somewhere (doc 02's third shard for redundancy, doc 03's N=5 option for quorum safety) — this is the first place in the series where a load-bearing number was carried forward with no safety factor questioned at all. That's worth flagging as a finding on its own, before even getting to what happens when it's tested.

---

## 4. Modeling hit ratio vs. cache size — the reassuring half of the finding

Real key-access patterns are heavily skewed (Pareto/Zipf-like), the same assumption doc 04's Section 3 warming approach leaned on. Using the standard 80/20 self-similar approximation (top 20% of keys by popularity account for 80% of requests, recursively), the cumulative hit ratio `H` as a function of the fraction `p` of the working set actually cached follows:
```
H(p) = p^k,  where k solves  0.8 = 0.2^k  →  k = ln(0.8) / ln(0.2) ≈ 0.1386
```

| Fraction of working set cached (p) | Hit ratio H(p) |
|---|---|
| 100% (full 500 GB) | 100% |
| 50% (~265 GB) | 90.8% |
| 25% (~132 GB) | 82.5% |
| 10% (~53 GB) | 72.7% |
| 1% (~5.3 GB) | 52.8% |

**The reassuring finding:** hit ratio degrades *gracefully*, not catastrophically, as cache memory shrinks relative to the working set. Cutting provisioned cache memory in half (100%→50%) only costs about 9 percentage points of hit ratio, and each further halving costs progressively *less* in absolute terms. Under this access pattern, being moderately under-provisioned on cache memory is a fairly safe failure mode by itself — worth stating as an honest, non-dramatic result, in contrast to the sharper cliffs found in the other three math passes.

---

## 5. The bottleneck this hides — DB load is not graceful in response to the same hit-ratio change

Section 4's degradation looks mild in percentage-point terms. But DB load isn't a function of hit-ratio percentage points — it's a function of `total_traffic × (1 − H)`, and `total_traffic` is 1,000,000/sec. Near H=95%, the `(1 − H)` term is small, which means **small absolute drops in H produce large *relative* swings in DB load**, precisely because Section 3 established the DB has no headroom to absorb any of it.

```
H     DB load = 1,000,000 × (1−H)     Nodes needed (÷15,000/node)
95%   50,000 /sec                      4
90%   100,000 /sec                     7    (2.0x the load, from a 5-point hit-ratio drop)
85%   150,000 /sec                     10   (3.0x the load)
80%   200,000 /sec                     14   (4.0x the load)
```

**Cross-referencing Section 4's table: a cache provisioned at only 50% of the working set (a plausible, not-even-extreme under-provisioning scenario) produces H≈90.8%, which drives DB load to roughly 2x its planned capacity** — even though the cache layer itself considered that same scenario a "graceful," fairly mild degradation. **This is the actual first bottleneck**, and it's a genuinely different shape than any prior doc's finding: the fragile component here isn't the cache degrading badly — it's the DB having been sized with a point estimate instead of a floor, so a mild, expected-range fluctuation in the cache's own (gracefully degrading) hit ratio still translates into a multi-x DB overload.

---

## 6. Iterate — provision the DB against a hit-ratio floor, not a point estimate

Rather than sizing the DB for the target hit ratio (95%) and hoping it holds exactly, size it for a **realistic worst-case floor** — say 85%, chosen as a plausible degraded state (roughly what Section 4's table shows for a cache running at half its intended memory footprint, a believable scenario during a partial outage, an under-provisioned rescale, or the deploy-time cold-start scenario from the companion warming doc):

```
DB nodes for a 85%-floor plan: 150,000 ÷ 15,000 = 10 nodes  (vs. 4 nodes for the point-estimate plan)
```

**6 additional DB nodes (4 → 10, a 2.5x increase) is the concrete cost of turning a fragile point estimate into a plan that survives a realistic hit-ratio dip without the DB falling over.** This is the same kind of explicit, costed trade-off as doc 03's N=3→N=5 decision (Section 5 of that math pass) — not free, and worth presenting as a deliberate choice rather than an automatic "always add more nodes" reflex. Whether 85% is the right floor to plan against (vs. 90% or 80%) depends on how confident the warming and stampede-protection mechanisms (companion docs) actually are at keeping hit ratio near target — this number is a planning input to negotiate, not a fact to discover.

---

## 7. Final sized design

| Layer | Naive first pass | After this pass |
|---|---|---|
| Cache nodes | 10 (throughput) / 8 (memory, value-only) | **10** — throughput-bound; per-key overhead correction (9 nodes) doesn't change the binding constraint |
| DB nodes | 4 (point estimate at exactly 95% hit ratio) | **10** (provisioned against an 85% floor, not a point estimate) |
| Real first bottleneck | — | **Not the cache — the DB's lack of headroom against its own hit-ratio assumption** |

**What this pass changed from the doc 04 design:** doc 04 stated a 95% hit ratio target and sized the DB to exactly match it, with the implicit (unstated) assumption that 95% is what will actually happen in practice. This pass is what shows that assumption was carrying real risk — not because the cache itself is fragile (Section 4 shows it isn't), but because nothing downstream of it had any tolerance for the cache being anything less than exactly on-target.

---

## 8. Follow-up: same procedure, different axis

- **Working set grows** (250M → 500M keys): recompute Section 1's memory-driven node count directly; unless it overtakes the throughput-driven count (10 nodes), the cluster size doesn't actually change — worth checking explicitly rather than assuming more data automatically means more cache nodes.
- **Traffic grows** (1M → 2M ops/sec): both the cache's throughput-driven node count *and* the DB's floor-based node count scale with it — recompute Section 1 and Section 6 together, since they share the same traffic input.
- **Target hit ratio itself changes** (e.g., a product decision to accept 90% instead of chasing 95%): re-run Section 5's table at the new target as the new "point estimate" row, then re-derive a new floor below *that* — the procedure is the same regardless of which specific percentage doc 04 had picked.
